# Configuration

The cluster and plugin configuration can be defined in multiple ways:

1. **Single file** — `template.yaml` contains everything (generated by [`init`](commands.md#init))
2. **Directory structure** — Configuration split across multiple files (generated by [`init --dir`](commands.md#init))
3. **Multi-environment** — Base configuration with environment-specific overlays (see [`env`](commands.md#env))

All approaches support the same configuration options. Use directory structure for complex setups, and multi-environment for managing dev/staging/prod configurations.

---

## Directory Structure (Recommended)

For complex configurations, use a directory structure to organize your configuration across multiple files:

```bash
# Generate directory structure
klastr init --dir --output my-cluster/
```

Generated structure:

```
my-cluster/
├── klastr.yaml          # Main configuration (name, provider, cluster)
├── plugins/             # Plugin configurations
│   ├── storage.yaml
│   ├── ingress.yaml
│   ├── cert-manager.yaml
│   ├── external-dns.yaml
│   ├── istio.yaml
│   ├── monitoring.yaml
│   ├── dashboard.yaml
│   └── argocd.yaml
├── apps/                # Custom application configurations
├── .env.example         # Environment variables template
└── README.md            # Documentation
```

### Loading Order

When using a directory, klastr loads and merges files in this order:

1. **`klastr.yaml`** — Main configuration (required)
2. **`plugins/*.yaml`** — Plugin configurations (alphabetical order)
3. **`apps/*.yaml`** — Custom application configurations (alphabetical order)

### Merge Behavior

- **Simple fields** (strings, numbers, booleans): Later files override earlier ones
- **Lists** (customApps, repos): Additive merge (items from all files are combined)
- **Objects** (plugins): Each plugin config is replaced entirely if redefined

### Example

**klastr.yaml:**
```yaml
name: production
provider:
  type: existing
  context: prod-cluster
cluster:
  version: v1.31.0
```

**plugins/monitoring.yaml:**
```yaml
plugins:
  monitoring:
    enabled: true
    type: prometheus
    ingress:
      enabled: true
      host: grafana.mycompany.com
```

**apps/my-app.yaml:**
```yaml
plugins:
  customApps:
    - name: my-app
      chartName: ./charts/my-app
      namespace: production
```

Usage:
```bash
# All commands work with directories
klastr lint --template my-cluster/
klastr run --template my-cluster/
klastr upgrade --template my-cluster/
```

## Single File Structure

The `template.yaml` file defines the entire cluster and plugin configuration. It is generated by the [`init`](commands.md#init) command and can be customized manually.

## Structure

```yaml
name: my-cluster              # Cluster name
provider:
  type: kind                  # Kubernetes provider
cluster:
  controlPlanes: 1            # Number of control planes
  workers: 2                  # Number of workers
  version: v1.31.0            # Kubernetes version
plugins:
  storage: { ... }            # StorageClass provisioner
  ingress: { ... }            # Ingress controller
  certManager: { ... }        # TLS certificate management
  externalDNS: { ... }        # Automatic DNS management
  istio: { ... }              # Service mesh
  monitoring: { ... }         # Prometheus + Grafana
  dashboard: { ... }          # Kubernetes dashboard
  customApps: [ ... ]         # Custom Helm charts
  argocd: { ... }             # GitOps with ArgoCD
```

## Base Fields

| Field | Type | Default | Required | Description |
|-------|------|---------|:---:|-------------|
| `name` | string | `my-cluster` | yes | Cluster name (used as kind name) |
| `provider.type` | string | - | yes | Kubernetes provider. Values: `kind` |
| `cluster.controlPlanes` | int | `1` | yes | Number of control plane nodes (min: 1) |
| `cluster.workers` | int | `2` | yes | Number of worker nodes (min: 0) |
| `cluster.version` | string | - | no | Kubernetes version (e.g., `v1.31.0`) |

## Plugins

Each plugin has an `enabled: true/false` field. Plugins not present in the YAML are considered disabled.

## Environment Variables

The `--env` flag (default: `.env`) loads environment variables from a file. Useful for secrets (e.g., SSH keys for ArgoCD).

`.env` file format:

```bash
ARGOCD_SSH_KEY="-----BEGIN OPENSSH PRIVATE KEY-----
...
-----END OPENSSH PRIVATE KEY-----"
```

Reference in template:

```yaml
argocd:
  repos:
    - name: private-repo
      url: git@github.com:user/repo.git
      sshKeyEnv: ARGOCD_SSH_KEY
```

## Templating

klastr supports Go templates in `template.yaml` for dynamic configuration:

```yaml
name: "{{ env "CLUSTER_NAME" "my-cluster" }}"
cluster:
  workers: {{ env "WORKERS" "2" }}
plugins:
  argocd:
    apps:
      - name: my-app
        values:
          image:
            tag: "{{ .GitCommit }}"
```

See [Templating Guide](templating.md) for complete documentation.

## Multi-Environment Configuration

For managing multiple deployment environments (dev, staging, production), klastr supports an overlay system similar to [Kustomize](https://kustomize.io/).

### Directory Structure

```
my-cluster/
├── klastr.yaml              # Base configuration (shared)
├── plugins/
└── environments/
    ├── dev/
    │   └── overlay.yaml     # Dev-specific patches
    ├── staging/
    │   └── overlay.yaml     # Staging-specific patches
    └── production/
        └── overlay.yaml     # Production-specific patches
```

### Base Configuration

The base `klastr.yaml` defines the common configuration shared across all environments:

```yaml
name: myapp
provider:
  type: kind
cluster:
  controlPlanes: 1
  workers: 1
  version: v1.31.0
plugins:
  storage:
    enabled: true
    type: local-path
```

### Overlay Configuration

Each environment has an `overlay.yaml` that patches the base:

```yaml
# environments/production/overlay.yaml
name: production
description: Production environment
base: ../../                    # Path to base configuration
patches:
  - target: name
    value: myapp-prod
  - target: cluster.controlPlanes
    value: 3
  - target: cluster.workers
    value: 5
  - target: plugins.monitoring.enabled
    value: true
  - target: plugins.certManager.enabled
    value: true
values:
  DOMAIN: example.com
  LOG_LEVEL: warn
```

### Creating Environments

Use the `env` command to manage environments:

```bash
# Create a new environment
klastr env create staging

# List all environments
klastr env list

# Show effective configuration
klastr env show production
```

### Using Environments

All main commands support the `--environment` (or `-E`) flag:

```bash
# Deploy specific environment
klastr run --environment dev
klastr run --environment staging
klastr run --environment production

# Validate environment configuration
klastr lint --environment production

# Upgrade environment
klastr upgrade --environment staging

# Check status
klastr status --environment dev
```

### Patch Syntax

Patches use dot-notation paths to target specific fields:

| Path | Example Value | Description |
|------|---------------|-------------|
| `name` | `"myapp-prod"` | Cluster name |
| `cluster.controlPlanes` | `3` | Control plane nodes |
| `cluster.workers` | `5` | Worker nodes |
| `cluster.version` | `"v1.31.0"` | Kubernetes version |
| `provider.type` | `"existing"` | Provider type |
| `provider.context` | `"prod-cluster"` | Kubectl context |
| `plugins.storage.enabled` | `true` | Enable storage |
| `plugins.ingress.enabled` | `true` | Enable ingress |
| `plugins.monitoring.enabled` | `true` | Enable monitoring |
| `plugins.dashboard.enabled` | `true` | Enable dashboard |
| `plugins.certManager.enabled` | `true` | Enable cert-manager |
| `snapshot.enabled` | `true` | Enable S3 snapshots |
| `snapshot.bucket` | `"my-backups"` | S3 bucket name |

### Environment Comparison

Example configurations for different environments:

| Setting | Dev | Staging | Production |
|---------|-----|---------|------------|
| **Workers** | 0 | 2 | 5 |
| **Control Planes** | 1 | 1 | 3 |
| **Monitoring** | No | Yes | Yes |
| **Cert-Manager** | No | No | Yes |
| **Domain** | dev.localhost | staging.example.com | example.com |
| **Log Level** | debug | info | warn |

### Best Practices

1. **Keep base minimal** — Include only truly shared configuration
2. **Use patches for differences** — Worker counts, feature flags, endpoints
3. **Use values for templating** — Domain names, resource limits, labels
4. **Version control** — Commit all environment configurations
5. **Document differences** — Use the `description` field in overlays
