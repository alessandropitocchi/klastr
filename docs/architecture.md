# deploy-cluster Architecture

## Overview

deploy-cluster is a CLI tool written in Go for provisioning local Kubernetes clusters with automatic plugin installation. The architecture is based on two main abstractions: **Provider** and **Plugin**.

## Project Structure

```
deploy-cluster/
├── cmd/deploy-cluster/          # CLI entrypoint
│   ├── main.go                  # Main
│   ├── root.go                  # Root command (cobra)
│   ├── init.go                  # init command: generates cluster.yaml
│   ├── create.go                # create command: creates cluster + plugins
│   ├── destroy.go               # destroy command: destroys cluster
│   └── get.go                   # get command: cluster/node/kubeconfig info
├── pkg/
│   ├── config/
│   │   ├── config.go            # Configuration structs and YAML parsing
│   │   └── env.go               # .env file loader
│   ├── provider/
│   │   ├── provider.go          # Provider interface
│   │   └── kind/
│   │       └── kind.go          # kind provider implementation
│   └── plugin/
│       ├── plugin.go            # Plugin interface
│       └── argocd/
│           └── argocd.go        # ArgoCD plugin implementation
├── example-app/                 # Example applications for ArgoCD
│   ├── app/                     # Plain YAML manifests
│   ├── nginx-helm/              # Example Helm chart
│   └── argocd/                  # Example Application manifest
├── cluster.yaml                 # Cluster configuration (generated by init)
├── .env.example                 # Example .env file for secrets
└── CLAUDE.md                    # Instructions for Claude Code
```

## Execution Flow

### `deploy-cluster init`

1. Generates a `Config` with default values
2. Serializes to YAML and writes to file

### `deploy-cluster create`

```
Load .env → Load cluster.yaml → Create cluster (Provider) → Install plugins
```

1. **Load .env**: Reads environment variables from `.env` file (optional)
2. **Load config**: Parses the `cluster.yaml` file
3. **Create cluster**: Calls the provider (kind) which:
   - Verifies kind is installed
   - Checks the cluster doesn't already exist
   - Generates the kind configuration (nodes, version, image)
   - Runs `kind create cluster` with streaming output
4. **Install plugins**: For each enabled plugin:
   - **ArgoCD**: Creates namespace → Applies manifest → Waits for ready → Adds repos → Creates Applications

### `deploy-cluster destroy`

1. Loads config or uses the name passed via flag
2. Calls `provider.Delete()` which runs `kind delete cluster`

## Interfaces

### Provider

```go
type Provider interface {
    Name() string
    Create(cfg *config.Config) error
    Delete(name string) error
    GetKubeconfig(name string) (string, error)
    Exists(name string) (bool, error)
}
```

Implementations: `kind` (current), `k3d` and `minikube` (future)

### Plugin

```go
type Plugin interface {
    Name() string
    Install(kubeconfig string) error
    Uninstall(kubeconfig string) error
    IsInstalled(kubeconfig string) (bool, error)
}
```

> Note: The ArgoCD plugin uses a specific signature `Install(cfg *config.ArgoCDConfig, kubecontext string)` to receive typed configuration. The interface may be standardized in the future.

## Configuration

Configuration is defined in `pkg/config/config.go` with the following main structs:

- **Config**: Root struct with name, provider, cluster, and plugins
- **ClusterConfig**: Topology (controlPlanes, workers, version)
- **ArgoCDConfig**: ArgoCD installation with repos and apps
- **ArgoCDRepoConfig**: Git/Helm repository with authentication (SSH, HTTPS)
- **ArgoCDAppConfig**: ArgoCD Application (Helm chart or Git path)

### Secret Management

Secrets (SSH keys, tokens) should not go in `cluster.yaml`. Two options:

1. **sshKeyFile**: Points to a local file (e.g., `~/.ssh/id_ed25519`)
2. **sshKeyEnv**: Reads from an environment variable, loadable from a `.env` file

The `.env` file is loaded automatically before config parsing.

## ArgoCD Plugin - Details

### Installation

1. Creates the specified namespace
2. Applies the official ArgoCD manifest with `--server-side --force-conflicts` (required for large CRDs)
3. Waits for the `argocd-server` deployment to be ready

### Repositories

Repositories are added as Kubernetes Secrets with the label `argocd.argoproj.io/secret-type: repository`. Supports:

- Public Git repositories
- Private Git repositories via SSH key
- Private Git repositories via HTTPS (username/password or token)
- Helm repositories
- Auto-detected `insecure` flag for non-HTTPS URLs

### Applications

Applications are created as ArgoCD `Application` CRD resources. Supports:

- **Helm chart**: From a public Helm repository with `chart` + `targetRevision`
- **Git repo**: From a Git repository with `path` + `targetRevision`
- **Values**: Inline in the configuration or from an external file (`valuesFile`)
- **Sync policy**: Auto-sync enabled by default with prune and selfHeal
